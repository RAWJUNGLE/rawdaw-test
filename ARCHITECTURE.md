# Архитектура RAWDAW Desktop

Документ описывает, как нативное C++ ядро встроено в приложение, какие были приняты решения и как при желании расширить интеграцию.

## Общая схема

```
┌─────────────────────────────────────────────────────────────┐
│ Electron — главный процесс (Node.js)                         │
│   src/main/main.js                                            │
│     require('native/rawcore')  ← C++ аддон (.node)            │
│     ipcMain.handle('rawcore:detectBpm', …)                   │
│     ipcMain.handle('rawcore:stretchOffline', …)              │
└───────────────▲───────────────────────────┬─────────────────┘
                │ IPC (structured clone)     │
┌───────────────┴───────────────────────────▼─────────────────┐
│ Electron — процесс рендерера (Chromium)                      │
│   src/main/preload.js                                         │
│     contextBridge → window.RawCore.detectBpm / stretchOffline │
│   renderer/index.html  ← ВАШ DAW, без изменений UI            │
│   renderer/native-hook.js  ← перехватывает _bdAnalyze         │
└──────────────────────────────────────────────────────────────┘
```

Главный процесс изолирован от рендерера (`contextIsolation: true`, `nodeIntegration: false`). Рендерер никогда не видит ни Node, ни нативный модуль напрямую — только узкий, замороженный объект `window.RawCore`, проброшенный через `contextBridge`. Все вызовы ядра — асинхронные, через `ipcRenderer.invoke`.

## Почему именно так

**Electron, а не Tauri.** Требование «UI и функциональность идентичны на всех ОС» — решающее. Electron везёт свой Chromium, поэтому Web Audio / AudioWorklet / WASM / DOM ведут себя одинаково на четырёх платформах. Tauri отдал бы рендеринг системным webview (WKWebView, WebView2, WebKitGTK), а они различаются — для DAW это риск рассинхронизации звука.

**Ядро в главном процессе, а не в рендерере.** Нативный аддон грузится в Node-процессе. BPM-анализ офлайновый (при импорте сэмпла), так что задержка IPC некритична, зато рендерер остаётся песочницей без доступа к ОС.

**N-API (node-addon-api), а не прямой V8.** ABI-стабильный интерфейс: один и тот же `.node` совместим между версиями Node, а под Electron модуль пересобирается через `@electron/rebuild`.

## Нативное ядро

### BPM (`native/rawcore/src/bpm.cpp`)

Алгоритм гибридный, потому что у одиночного детектора и у автокорреляции разные слабые места:

1. **Основная оценка — автокорреляция огибающей онсетов.** Считаем спектральный поток (STFT, окно 1024, шаг 512, окно Ханна, положительные приращения магнитуд по нижней четверти спектра), вычитаем среднее и выпрямляем. По огибающей берём автокорреляцию в диапазоне лагов для [bpmMin, bpmMax] и выбираем фундаментальную частоту с предпочтением по сумме гармоник (+0.5·acf[2·lag]+0.33·acf[3·lag]), чтобы не залочиться на удвоенном/утроенном темпе. Параболическая интерполяция пика даёт точность до долей BPM. **Это возвращаемое значение `bpm`.**
2. **Кросс-проверка — BeatDetektor** (ваш `core/BeatDetektor.*`), запущенный ровно как в эталонном `main.cpp`: на каждый кадр 1024 сэмпла — моно-даунмикс, реверс-бинарное переупорядочивание, `DanielsonLanczos<512>.apply`, схлопывание в первые 256 + последние 256 бинов, затем `det.process` → `contest.process` → **`contest.run()`**. Короткий материал зацикливается до ≥40 c, чтобы contest успел сойтись. Возвращается как `bpmRaw`.
3. **Фаза долей** — гребенчатая кросс-корреляция огибающей на выбранном периоде; массив `beats[]` с временами долей.
4. **Уверенность** — резкость пика автокорреляции, +0.15 если BeatDetektor согласен в пределах октавы.

Тонкости, важные для повторной сборки (заложены в код, не регрессировать):
- `FFT.h` использует `M_PI` и `sin()` в шаблонах, но сам не включает `<cmath>` — обёртка подключает их **до** включения `FFT.h`. Сам вендоренный файл не тронут.
- `current_bpm` в BeatDetektor — это **интервал в секундах**, BPM = 60/current_bpm.
- `win_bpm_int` выставляет именно `contest.run()`, а не `process()`. Без `run()` победитель не определяется.
- Конструктор `BeatDetektor(MIN, MAX)` требует `MAX ≤ MIN·2−1` (одна октава). Объединение нескольких октавных детекторов в один contest даёт ложный лок с полной уверенностью — поэтому за основу взята автокорреляция.

### Стретч (`native/rawcore/src/stretch.cpp`)

Обёртка над **Signalsmith Stretch** (`signalsmith-stretch.h`, как есть). Зависимость `signalsmith-linear` завендорена в `native/rawcore/vendor/signalsmith-linear/`. Режимы:

- **Равномерный**: `timeRatio` (длина выхода/входа) + `semitones` (питч). Используется `stretch.exact(...)`.
- **Warp-маркеры**: пары `{inputBeat, outputBeat}` переводятся в сэмплы через `secondsPerBeat` и сортируются по выходу; буфер обрабатывается посегментно с локальным коэффициентом растяжения для каждого сегмента (`stretch.process` сохраняет непрерывность между вызовами). Логика повторяет `computeStretchSegments` / `clipOutputBeatToInputBeat` из рендерера.

Портируемость: вендоренный `fft.h` использует `std::memcpy` без `<cstring>` — обёртка подключает его первым.

## Интерфейс N-API (`native/rawcore/src/addon.cpp`)

```
detectBpm(channels, sampleRate, opts?)
    channels:   Float32Array[] (по каналу) либо один Float32Array (моно)
    opts:       { bpmMin=80, bpmMax=180 }
    → { bpm, bpmRaw, confidence, beats[] }

stretchOffline(channels, params)
    params:     { sampleRate, timeRatio=1, semitones=0,
                  secondsPerBeat=0.5, markers:[{inputBeat,outputBeat}] }
    → { channels: Float32Array[], frames }
```

## Аддитивный хук рендерера

`renderer/native-hook.js` — классический скрипт, подключённый перед `</body>`, после основного скрипта DAW. В классических скриптах объявления верхнего уровня попадают в глобальную область, поэтому хук может переопределить `window._bdAnalyze` — единственную точку входа анализа BPM в проекте (её зовёт `ceEnsureBpm`).

Логика хука:
1. Если `window.RawCore` нет (обычный браузер) — выходим, ничего не меняя.
2. Иначе оборачиваем оригинальный `_bdAnalyze`: сначала вызываем его (он даёт `transients` для сетки warp и служит гарантированным запасным вариантом), затем зовём `RawCore.detectBpm` и **заменяем только число `bpm`**, сворачивая его в ту же октаву (55–200), что и оригинал.
3. Любая ошибка нативного вызова → молча остаётся JS-результат.

Так гарантируется: ни один видимый элемент не добавлен и не изменён; транзиенты, маркеры, кэш — идентичны; в браузере поведение ровно прежнее.

## Возможные расширения (вне текущих рамок)

**Нативный стретч в экспорте.** Сейчас экспорт и реалтайм используют существующий WASM-путь Signalsmith — это сохраняет идентичный результат. Чтобы перевести офлайн-рендер клипов на `window.RawCore.stretchOffline`, нужно в функции рендера клипа (`deviceOffline` / путь стретча в `initStretch`) заменить вызов WASM на IPC-вызов ядра и собрать сегменты из существующего `computeStretchSegments`. Это меняет код экспорта, поэтому требует отдельной проверки на побитовую совместимость — отсюда вынесено в опциональный шаг.

**superduper-dsp (Rust, CLAP).** Это набор эффектов в формате плагина CLAP. Интеграция возможна двумя путями: (а) собрать как CLAP-плагин и грузить хостом плагинов; (б) скомпилировать DSP-функции в статическую библиотеку Rust и пробросить через ещё один N-API аддон. Оба — отдельные сборочные конвейеры (нужен Rust toolchain), в порт ядра DAW не входят.

**Just-a-Sample (JUCE C++).** Полноценный инструмент-плагин с собственным CMake-проектом и релизными скриптами под три ОС. Собирается независимо как VST3/AU/standalone средствами JUCE; в N-API ядро его встраивать нецелесообразно. Документируется как отдельный артефакт.
